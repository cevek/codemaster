## Специфика codemaster

**Гейт:** `npm run fix-and-check` (EXIT 0) — авто-сносит механический dead-code (unused imports через
eslint, truly-dead exports через knip; не удаляй руками). Полный `npm test` — один раз в конце.

**Экономь прогон** (это батарея и время юзера): один раз редиректни `npm test >/tmp/t.txt 2>&1` и
грепай ФАЙЛ сколько надо; для итерации — точечно `node --test test/foo.test.ts`, не весь suite.
Сначала глянь реальный выхлоп упавшего теста, потом чини точечно, не вслепую.

**Полный сьют гоняй РЕДКО — только когда его результат реально мог измениться (батарея/CPU юзера).**
Правило: полный `npm test` — РОВНО ОДИН РАЗ, на DONE, на финальном состоянии, что отдаёшь. НЕ гоняй
его повторно в итерации — итерация идёт на таргетных `node --test test/foo.test.ts` (дёшево, без
лока) или на ЛЁГКОЙ группе `npm run test:light` (unit+golden, ~12с) для широкой-но-дешёвой проверки.
Сначала глянь выхлоп упавшего теста (`npm test >/tmp/t.txt 2>&1`, грепай ФАЙЛ), чини точечно.
Тест-группы по стоимости: `test:light` (unit+golden, ~12с — inner-loop); `test:heavy`
(e2e+differential — real-spawn плюс cold-Program-оракулы, ~90% времени); `test:e2e` / `test:differential`
точечно; `npm test` = всё (авторитетно, на DONE). Профиль времени — `npm run test:timing` → test-timing.html.
**Твой DONE-прогон — ЭТО верификация: менеджер ему ДОВЕРЯЕТ (сам полный сьют перепрогоняет редко).**
Не халтурь — прогони реально, на финальном коммите, зелёным; false-green всплывёт позже и дороже.

**После ребейза (менеджер попросил на мердж) НЕ перегоняй полный сьют вслепую.** Ребейз чистый (0
конфликтов) И входящие изменения disjoint от твоей edit-поверхности (менеджер режет треки edit-disjoint) →
твой ДОнеймовый green ВСЁ ЕЩЁ держится, повторный полный прогон — мусор. Что реально нужно после ребейза:
прогони `fix-and-check` (ловит prettier-дрейф — rebase ПРОПУСКАЕТ prettier-хук; закоммить формат-дельту)
и доложи «rebase clean, prior green держится». Полный `npm test` перегоняй ТОЛЬКО если ребейз был с
КОНФЛИКТАМИ или входящее реально трогает файлы/поведение, от которых зависит твой код. Сомневаешься —
скажи менеджеру, пусть решит. Авторитетную верификацию на main менеджер гонит сам.

**Тест-лок (несколько воркеров на одной машине).** Два одновременных полных сьюта трэшат CPU
суперлинейно. Любой ТЯЖЁЛЫЙ прогон — полный `npm test` ИЛИ `fix-and-check` — ТОЛЬКО через machine-wide
**мьютекс K=1**: `node ~/.codemaster-orch/with-test-lock.mjs -- npm test` (и `... -- npm run
fix-and-check`). Другой держит лок → ждёшь в очереди (норма, не гоняй голый сьют). Таргетные
`node --test` — без лока. (Лок провижнит менеджер; нет helper'а — пингани, не гоняй параллельно.)

**Oracle-backed тест (§16):** независимый оракул — cold `ts.Program` / cold reparse / отдельный
код-путь; НЕ grep и НЕ golden-only; реально падает ДО фикса.

**Соответствие проекту:** главный инвариант (never-lie / never-hang, ARCHITECTURE §1/§3), лейеринг
(downward-only, `common/`→`core/`, plugin-DAG), файл ≤300 строк, внешние вызовы обёрнуты →
`ToolFailure`, boundary zod-валидирован. Source-of-truth: CLAUDE.md · ARCHITECTURE.md · CONTRIBUTING.md
· src/README.md.

**Ревьюверы под тип правки** (поднимаешь как субагентов): `bug-reviewer` — всегда; `architecture-reviewer`
— новый seam / кросс-модуль / структура; `copy-paste-reviewer` — заметный новый код; `doc-sync-reviewer`
— правка меняет публичный контракт / output-surface / доки.

**Risk-остриё codemaster для брифа ревьюверу:** stale/freshness (§3.5), proof-span off-by-one
(1-based Loc ↔ 0-based offset), мисидентификация (алиас-импорт / same-name символ), silent-truncation,
необёрнутый внешний вызов, блокировка оркестратора синхронной тяжёлой работой (§2).

**Backlog** — task-manager (MCP), не руками. **Догфуд:** символьные вопросы веди через codemaster MCP;
его баг/трение подай в его `feedback` сам.
