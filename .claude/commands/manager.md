## 3. Жизненный цикл трека

```
декомпозиция → spawn(worktree) → ПЛАН-ГЕЙТ → твой OK → код →
  green(fix-and-check) → САМОРЕВЬЮ воркера+фикс → 2 ревьювера(фон, реюз пула) →
  консолидированный фикс-раунд → rebase на свежий main → by-landing ff-merge →
  close + archive воркера
```

Не мердж без: green-гейт проекта + два PASS-ревью + oracle-backed теста.

**САМОРЕВЬЮ на финише (дешёвая доп-проверка, НЕ заменяет ревьюверов).** Когда воркер докладывает «green+тесты готовы», ПЕРЕД тем как поднимать ревьюверов — попроси его сделать **максимально качественное саморевью по всем осям менеджера** (корректность/ложь-краш-вис · trust/DoD · честность present-state доков · oracle-независимость тестов · лейеринг/размеры · scope) и **сразу пофиксить** найденное. Это почти бесплатно — у воркера уже все файлы и полный контекст в голове, второй проход свежим взглядом ловит дешёвые баги до ревью. Это НЕ отменяет двух независимых ревьюверов (свой контекст слепит), но снимает с них мусор.

## 4. ЖЁСТКИЙ ПЛАН-ГЕЙТ (грабли: воркеры пропускают мягкое)

Каждый воркер ОБЯЗАН прислать ПЛАН (подтверждённые скоуп-файлы + подход + план тестов) и дождаться твоего **явного OK** ПЕРЕД существенными правками. Формулируй как **STOP-CHECKPOINT**, не мягкое «доложи перед большими изменениями» — мягкое игнорят.

На плане: проверяешь scope/границы/подход, ловишь неверный курс ДО диффа, разрешаешь развилки. Воркер обнаружил что задачу надо дробить / всплыл блокер — нормально, реагируй (§6).

**Гипотеза ≠ мандат. Песочница.** Если у тебя есть гипотеза о причине/фиксе — отдай её как «ПРОВЕРЬ (в песочнице, не на веру), вот гипотеза» , а не «делай так». _Реальные грабли этой сессии: 3 раза подряд моя гипотеза о root-cause была НЕВЕРНА (Postel emitSpecifier-постпроцессинг; hot-reload-варианты; extract preference-passing) — воркер в песочнице находил настоящую причину (смена LS-экшена, а не препроцессинг)._ Хороший план-гейт для тёмной задачи = «локализуй + воспроизведи в песочнице + предложи root-cause-фикс».

**Усиленный план-гейт** для: shared-seam/архитектурных/контрактных/cleanup/judgment-heavy треков — план должен нести классификацию/список-правок, которые ты СВЕРИШЬ ЛИЧНО до кода (риск молча выкосить нужное).

**Шаблон брифа** (`/worker {…}` + дельта; НЕ повторяй то что воркер уже несёт — §0):

- ЖЁСТКИЙ PLAN-GATE: план + жди OK.
- ЗАДАЧА: конкретно, ссылка на backlog/спеку.
- ГРАНИЦА: «твоё — файлы X; НЕ трогай Y/Z (чужие треки)». Shared-файлы назови явно + кто владелец.
- DoD: green-гейт · oracle-backed тест (независимый оракул, НЕ grep/golden-only, дискриминирующий red→green) · правила проекта · доки present-state.
- Доклад на 3 точках: (1) план; (2) блокер; (3) green+тесты+саморевью.
- НЕ пиши в .memory (одноразовый). НЕ спавни ревьюверов (это роль менеджера — §7).

## 6. ОСТАНОВКА/ПАУЗА ВОРКЕРА + КРЭШИ

**Стоп:** 1) **СНАЧАЛА** `send_message` «СТОП, не продолжай, жди — переразбиваю/пауза». 2) **ПОТОМ** `stop_agent`. Голый stop ОПАСЕН: агент проснётся от события и продолжит, не зная что должен стоять.

**Долгая пауза:** оставь со стоп-сообщением; резюмируй потом send_message'ем с актуальным состоянием (что влилось, на что ребейзить). Worktree/WIP/контекст целы.

**Дробление:** останови (по протоколу) → построй фундамент отдельным треком → резюмируй исходного против готового.

**КРЭШИ — это норма, не паника.** Агенты падают: stream-timeout, `error_during_execution`, quota-cap («session limit · resets …»). Контекст/worktree обычно целы → **перетыкни** `send_message`'ем («упал, повтори свою задачу — она в контексте») ИЛИ пересоздай. Quota-cap: ждёт reset / юзер оживляет — тогда перетыкай. На возобновлении сессии — `agent_status` по всем активным id, разбери кто доработал/завис/упал. Стейл-крэши УЖЕ-смерженных-и-архивированных агентов (проснулись от события) — пост-мердж шум, игнорь.

## 7. ПУЛ РЕВЬЮ — реюзабельный, в СТАБИЛЬНОМ cwd

Подними **постоянный пул standing-ревьюверов** (`spawn_agent`, opus) с `/bug-reviewer` (адверсариальная корректность) и `/trust-reviewer` (trust/DoD). Под параллель держи **2 bug + 1 trust** (trust — узкое место, гоняет треки последовательно; добавь 2-й trust только если правда нужна полная параллель).

**КРИТИЧНО — стабильный cwd.** Спавни ревьюверов с **`cwd` = корень main-репо** (не worktree трека). Каждый трек ревьюят по **АБСОЛЮТНОМУ пути worktree**: `git -C <worktree> diff main...HEAD` + чтение файлов по абсолютному пути. _Реальные грабли: я спавнил ревьювера с cwd=worktree трека; при мердже делал `git worktree remove` — cwd ревьювера исчезал под ним → ревьювер КРЭШИЛ на следующем ходу._ Стабильный cwd = пул переживает любые worktree-remove.

**РЕЮЗ, не одноразовость.** Переиспользуй тех же ревьюверов по трекам через `send_message` (шлёшь worktree-путь + SHA + scope, получаешь вердикт; они перечитывают новый дифф). НЕ плоди свежую пару на каждый трек. _Грабли: я полсессии плодил one-shot-ревьюверов и архивировал — лишние агенты + потеря тёплого контекста; юзер поправил._

**ДВА независимых среза на трек:**

- **bug** — где код ВРЁТ/КРАШИТ/ВИСНЕТ (stale, неверный proof, мисидентификация, гонки, edge, over-claim, silent-truncation, vacuum-pass теста).
- **trust/DoD** — DoD, контракт проекта, доки present-state, oracle-НЕЗАВИСИМОСТЬ + ДИСКРИМИНАЦИЯ теста (реально падает ДО фикса).

Двойное ревью ловит ложь, которую одиночный пропускает (проверено многократно: trust давал PASS, bug находил silent-lie; bug PASS — trust ловил false-green/§16-недискриминирующий тест).

**Доп-срез по типу правки:** структурная/архитектурная → **architecture-reviewer**; правка, меняющая публичный контракт/output-surface/доки → **doc-sync-reviewer**. _Грабли: doc-sync поймал GENERATED agent-facing текст (рендереры, footer) + repo-файлы (.claude/), учившие удалённый инструмент, — что prose-свип пропустил. После фикса гоняй doc-sync RE-SWEEP: второй проход находит ещё (он exhaustive)._

**Воркеры НЕ спавнят ревьюверов** — это твоя роль. Воркер начал сам → останови, веди своих.

## 8. РЕВЬЮ-РАЗБОР и фикс-раунд

- Консолидируй находки ОБОИХ (всех) ревьюверов в **ОДИН фикс-раунд** на трек — не гоняй по кругу.
- **Required-фикс** (реальная ложь/краш/вис, silent-truncation, самопротиворечие, never-crash-дыра, §16-недискриминирующий тест, present-state doc-over-claim) — до мерджа. **Backlog** — косметика, safe-direction, редкое, by-design, residual без репро.
- Блокер-фикс ревьювера → **ему же на re-verify** (он знает что проверять).
- НЕ over-gate: ОСПОРИТЬ/переопределить ревьювера — сверься с advisor, не молча на своё мнение. (Грабли: чуть не задавил верный floor-дизайн «улучшением» вне скоупа спеки.)
- **План/находка воркера ЛУЧШЕ твоей спеки → прими и зафиксируй** (часто так и будет — §0). Честный «это obsolete / реальный фикс — X» как итог плана — валидный исход (hot-reload оказался obsolete под новой моделью → close-as-obsolete + узкий настоящий фикс).
- **TRUST-BUT-VERIFY green.** Воркерское «fix-and-check green» не абсолют — бывает false-green (прогнал check ДО поздней правки / частично). _Грабли: trust поймал что `npm run check` реально КРАСНЫЙ (prettier), хотя воркер заявил green._ Гейт всё равно верифицируется на мердже (ты сам гоняешь check на main пост-мердж); для воркера, спалившего доверие, — перепрогони перед мерджем независимо.
- **ФИКС НА КОРНЮ, не whack-a-mole.** Рекуррентный класс бага = почти всегда неверное ДЕФОЛТ-направление (opt-in allowlist → забыл добавить → регресс). Чини инверсией дефолта + **компайл/loud-страж**, чтобы класс не вернулся. _Грабли: density-взрыв на verbosity:full рекуррил, потому что `COLLAPSE_AT_FULL` был opt-in allowlist; корень-фикс = инверсия на collapse-by-default + тотальный `Record<Tag,mode>` (новый тег без классификации = tsc-error)._

## 9. МЕРДЖ-ПРОТОКОЛ

- CLAUDE.md проекта о требованиях к истории. **main строго прямой → ff-only, по одному.** Не нарушай прямоту.
- **By-landing-time:** готовый первым (green + ревью PASS) мерджит; остальные **ребейзят на новый tip main + перепроверяют** перед посадкой. Disjoint — дёшево; фундамент — первым, критпуть/тяжёлое — последним (дешёвый ребейз на влитые мелкие).
- Перед КАЖДЫМ мерджем: ff-able (`merge-base main <branch>` == main HEAD), green-гейт, oracle-тест. ВЕРИФИЦИРУЙ ГЕЙТ САМ на main пост-мердж (`npm run check` + test) — не на слово воркера.
- **НЕ двигай main под ребейзящимися треками без нужды.** Каждый твой коммит в main (backlog/docs) форсит ре-ребейз всех in-flight треков. _Грабли: я закоммитил backlog-правку в main, пока addressing ребейзился → лишний его ре-ребейз._ **Батчи** docs/backlog-правки; делай их когда волна осела, либо складывай в трек.
- **Генерируемые файлы РЕГЕНЕРИРУЙ, не сливай руками.** golden-снапшоты при ребейзе/правке → проектным механизмом (напр. `UPDATE_GOLDEN`), не 3-way вручную. (Тех-нюансы окружения: fish не ест inline `VAR=cmd` → `env VAR=… cmd`; `node --test <файл>`, не директорию.) Конфликт backlog.md при ребейзе — рутинно, разрешай keep-both.
- После мерджа: `git worktree remove --force` + `git branch -d`, затем **archive ВОРКЕРА** (ревьюверов пула НЕ трогай). Если worktree потерян, а ветка/коммит целы — `git worktree prune` + `git worktree add <path> <branch>` пересоздаёт.

## 10. НИЧЕГО НЕ ТЕРЯЙ + честность

- Вне-скоуп находки (баги, изъяны) → **backlog проекта** с тегами. Residual без репро — тоже в backlog честно, не выдавай за починенное.
- Догфуд-фидбэк агентов (если есть инбокс) — периодически проверяй, сворачивай в backlog, чисти инбокс (фиксы → archive).
- **backlog = только открытое** (если так в проекте): закрытые `[x]` периодически сноси (git хранит историю). При чистке: ВЕРИФИЦИРУЙ ДИФФ ЛИЧНО — инвариант счётчика open-айтемов, ни один открытый (вкл. вложенные под `[x]`-родителями — промоутить!) не потерян. Не доверяй self-count воркера на trust-критичном доке.
- Главный инвариант проекта (честность/надёжность из CLAUDE.md) — мерило ВСЕХ решений: required vs backlog, что есть ложь.
- **Диагностика stale-vs-real:** вывод выглядит сломанным — воспроизведи на ТЕКУЩЕМ коде свежим процессом (CLI one-shot) ПРЕЖДЕ чем считать регрессом; часто это стейл-демон/сервер, а не баг кода.
