# t-679091 — plan (semantic fan-out pre-warm guard)

Refuse a genuinely-heavy SEMANTIC fan-out op on an oversized **in-process** repo with an honest,
actionable `ToolFailure` (redirect to `isolation:'process'`) instead of warming into an uncatchable
OOM that kills the singleton daemon. The direct analogue of the accepted t-333163 search_symbol
size-guard; §1 "never crash", not "always succeed".

## Problem (current main)
t-000052 shipped the process-mode MECHANISM (per-engine `--max-old-space-size`, deadline→SIGKILL
child, child-exit→`ToolFailure{oom}`, respawn) but it is OFF by default: `isolation` defaults to
`in-process` (host-build.ts:41, ARCHITECTURE §2). Fix A pruning + the t-333163 guard cover only
`search_symbol`; the semantic ops are explicitly NOT guarded. So under the default config a heavy
`find_usages` on a references-monorepo (where Fix A's discovery pruning does NOT subsume → full
LS fan-out across all programs) still OOMs → kills the in-process daemon (uncatchable, §1).

## The isolation seam (the one new piece of plumbing)
The guard must fire ONLY in-process (process-mode already survives via t-000052). The op runs INSIDE
the engine, which today does NOT know its isolation mode (`ProjectHost.isolation` exists at
host.ts:25 but lives orchestrator-side; the process-mode engine is in a forked child with no host
handle). The engine's mode is fixed by WHICH construction path built it:
- `host-build.ts:68` (in-process branch) → always in-process.
- `engine-child.ts:67` (forked child) → always process.

So thread it explicitly:
1. `EngineDeps` (engine.ts:38) gains `isolation?: 'in-process' | 'process'` (optional, default
   `'in-process'` — backward-compatible for any stray direct-createEngine test call).
2. host-build.ts passes `isolation: 'in-process'`; engine-child.ts passes `isolation: 'process'`.
3. `buildDaemonInfo` (request-helpers.ts:61) copies it into the op-context `DaemonInfo`
   (ops/registry.ts) → every op reads `ctx.daemon.isolation`. Consistent with the orchestrator-side
   `ProjectHost.isolation` (same two sites set both), just engine-visible.

## The guard (shared helper)
New `src/ops/guard/semantic-fanout-guard.ts` exporting a pure check:
`semanticFanoutRefusal(ctx, ts, force): ToolFailure | undefined` —
refuse (return a ToolFailure) iff ALL of:
- `ctx.daemon.isolation === 'in-process'` (process-mode → never refuse), AND
- `force !== true` (per-call override, mirrors the size-guard), AND
- `ts.estimateSourceFileCount()` is `ok` AND `> ts.searchWarmMaxFiles` (the SAME cheap git-source
  count + threshold the size-guard uses — no program build, no LS warm). An estimate FAILURE (git
  hiccup) falls THROUGH → no refusal (an optimization, never over-refuse — mirrors t-333163).
Message: actionable, e.g. `repo too large (N source files > threshold) for in-process semantic
fan-out — set daemon.isolation:'process' in codemaster.config (or pass force:true to warm anyway)`.
Distinct from the size-guard's message (which redirects to symbols_overview/syntactic — a discovery
alternative that does NOT exist for find_usages; here the only escape is process-mode or force).

Applied at the VERY TOP of each guarded op's `run()`, BEFORE any resolve/warm (a name-addressed
find_usages warms the LS inside `resolveByName`→searchSymbols, so the guard must precede resolve).

## Scope — which ops (⚠ SCOPE CORRECTION vs the task's pinned three — see below)
The addressing mode matters, not just the op:
- `find_usages` / `impact` / `importers_of` — fan the REFERENCE search across all programs
  REGARDLESS of addressing (`findReferencesAcross` / `programsContaining` builds programs to test
  `containsFile`) → guard UNCONDITIONALLY.
- `find_definition` — cheap for symbolId / file+line+col / name+file (single containing-program
  resolve, `sourceFileAcross`, primary-first + lazy), BUT a **bare-`name`** target resolves via
  `resolveByName → searchSymbols`, which fans navto across ALL programs when Fix A's pruning does NOT
  subsume (the references-monorepo case) = the ORIGINAL 6.1 GB build-every-program OOM of t-167395.
  Evidence: resolve-target.ts:151 (`resolveByName`→`searchSymbols`) + search.ts (`active = programs`
  when `!covered`). So guard `find_definition` IFF **bare-name** (`args.name` set, `args.file` AND
  `args.symbolId` unset) — the op already computes this exact predicate (`nameOnly`,
  find-definition.ts:45). The cheap addressing paths stay cheap (never refused).
Do NOT guard:
- `search_symbol` — already has its own guard (t-333163).
`force:true` bypasses everywhere (add the `force` arg to schemas that lack it —
impact/importers_of/find_definition likely need it).

**Why this corrects the pinned scope:** the task pins "find_usages/impact/importers_of, NOT
find_definition (cheap)". That is right for 3 of find_definition's 4 addressing modes but WRONG for
bare-name, which shares the exact searchSymbols all-program fan that is the t-167395 crash. Excluding
find_definition wholesale leaves `find_definition {name:"Foo"}` OOM-killing the daemon — the very
symptom this task closes. Flagged as open question 3; needs manager sign-off before code.

## Config
Reuse `ts.searchWarmMaxFiles` (default 4000) — same estimator, same LS-warm OOM surface, one knob.
(A sibling `ts.semanticFanoutMaxFiles` is deferred unless calibration shows the semantic fan-out
OOMs at a materially different count than navto — flagged as an open question, not built now.)

## Discriminating tests (test/differential + a unit test)
1. **find_usages refused in-process over threshold** — fixture with config `ts.searchWarmMaxFiles: 0`
   (any repo exceeds) → `find_usages` returns a ToolFailure whose message names `isolation:'process'`
   (NOT an OOM, NOT a result). Same for `impact`, `importers_of`.
2. **find_definition addressing-mode discriminator (THE key test)** — same oversized in-process
   fixture: `find_definition {symbolId | file+line+col | name+file}` NOT refused (cheap paths stay
   cheap) vs `find_definition {name}` (bare-name) IS refused with the isolation:'process' redirect.
   This pair discriminates the bare-name predicate — the real logic (not "never refused", which would
   mutation-test code never written).
3. **force:true bypasses** — `find_usages {force:true}` over threshold → runs (returns usages).
4. **process-mode never refuses (unit test of the helper)** — `semanticFanoutRefusal` with
   `isolation:'process'` over threshold → returns `undefined` (E2E process-mode is heavier; the
   isolation gate is unit-tested directly). Mutation-verify: drop the isolation check → returns a
   refusal → test fails.
5. **estimate failure falls through** — a failing estimator (git seam) → no refusal (no over-refuse).
6. Below threshold → normal behavior unchanged (regression guard).
