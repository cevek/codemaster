# t-167395 — plan (Fix A discovery-pruning + backstop split)

Present-state plan for eliminating the name-addressed OOM. Two independent pieces; Fix A is
daemon-disjoint, the backstop depends on t-000052 (process isolation).

## Problem (confirmed, current main)
`plugins/ts/search.ts::searchSymbols()` loops `host.programs()` and calls
`getProgram()`+`getNavigateToItems()` on EVERY loaded program → forces all ~25 sibling/member
programs to build in one heap on backoffice2 (6082 ts/tsx, loose root globs whole repo). Peak
~6.1 GB / ~40 s → hard V8 OOM → in-process singleton daemon dies (uncatchable, §1 violation).

Current mitigation already merged (t-333163 size guard): default `search_symbol` on a repo above
`ts.searchWarmMaxFiles` (4000) REFUSES to warm and redirects. So backoffice2 default no longer
OOMs — it refuses. The OOM path today is only `force:true` (and any raise of the threshold).

---

## Fix A — discovery-only program pruning (SAFE, daemon-disjoint)

### Design (airtight, all-or-nothing)
navto discovery is resolution-independent: a file's named declarations come from parse+bind of
that file; `paths`/`baseUrl` do not change what a file DECLARES. So a program whose contribution
is already scanned adds zero new symbols (and we already dedup by `fileName|textSpan.start`).

Formulation (chosen over a per-program glob-subset test, which has a real hole — primary globs
P's roots but not a file G reached only via P's unique `paths` resolution):

> In `searchSymbols`, navto the PRIMARY first (already primary-first order). Then prune ALL
> remaining programs **iff** the primary's built-in-root project source-file set
> (`primary.getProgram().getSourceFiles()`, filtered to in-root non-junk with the existing
> `rootPrefix` + `isJunkRelPath` logic from single.ts) **⊇ the in-root git TS surface** — the
> exact set `estimateSourceFileCount`/`gitSourceFilesSync` + `isScannedSourcePath` already list.

**Why provably complete for navto:** any symbol a sibling could surface comes from a source file
that is (a) in-root — out-of-root is the disclosed t-515730 caveat, git-invisible at root — and
(b) non-junk → therefore in the git surface → therefore (under the ⊇ test) already a primary
source file → already scanned. All-or-nothing is correct because the OOM case IS the all-covered
case (loose root). On a proper `references` monorepo the primary does NOT subsume the surface →
test fails → prune nothing → full fan-out unchanged → the genuinely-heavy case falls to the
backstop.

**Safety boundary (the §3.4 line):** the change lives ENTIRELY in `searchSymbols`. Every caller
is discovery-only: `resolveByName` / `resolveAllByName` / rebind (resolve-target.ts:148/188/265)
and the `search_symbol` op. `find_usages`'s REFERENCE fan-out uses `programsContaining` /
`findReferencesAcross`, NOT `searchSymbols` — it uses searchSymbols only to LOCATE the
declaration, then the LS re-resolves references across all containing programs. So pruning cannot
leak into references; the resolution-program requirement for find_usages is untouched.

### Files
- `src/plugins/ts/search.ts` — the pruning loop in `searchSymbols` (primary-first, accumulate
  primary source-file set, compute the ⊇-surface gate once, skip the rest when covered).
- `src/plugins/ts/ls-host.ts` — likely a small seam so `searchSymbols` can get the in-root git
  surface set + the primary program without a new repo walk (reuse `gitSourceFilesSync` /
  `isScannedSourcePath`; primary via `host.programs()[0]`). Keep it a pure read helper.
- No change to resolve-target.ts callers (they consume `searchSymbols` output unchanged).

### Guard reconciliation (touches t-333163 territory — may need separate GO)
Post-Fix-A, peak memory is bounded by the LARGEST SINGLE program (~primary), not the sum. The
t-333163 size guard counts the TOTAL surface, which after Fix A measures the wrong thing (a
6082-file single primary is fine). Sequence explicitly:
1. Fix A → makes `force:true` safe + delivers the measurement (0.95 GB / 4 s vs 6.1 GB / OOM).
2. Guard follow-on → gate on post-pruning PEAK (single-program size), not total surface, so
   default `search_symbol` works on a loose-root monorepo instead of refusing. This edits
   `surface-size.ts` / `search-symbol.ts` (t-333163's code) → flag as a distinct sub-step; do
   NOT claim Fix A "fixes backoffice2 default" — it does not until the guard moves.

### Repro + measurement
`node src/bin.ts op search_symbol '{"name":"VirtualSetup","force":true}' --root
/Users/cody/Dev/backoffice2` with `NODE_OPTIONS=--max-old-space-size=12000` (avoid MCP — a crash
kills the shared daemon). Before: ~6.1 GB / ~40 s. After: ~0.95 GB / ~4 s. CLI one-shot only.

### Oracle — completeness (byte-identical)
On a repo where the loose root subsumes members, the pruned `SearchView` must equal the
full-fan-out `SearchView` EXACTLY — `matches` (ordered), `total`, `filteredOutByPath`. Compare
pruned vs `force`-full on backoffice2 (78 matches for VirtualSetup) and on an inline fixture.

### The one discriminating test (separates sound design from the naive-buggy one)
Inline VFS fixture: a PARTIAL-glob primary (NOT whole-repo — e.g. `include:['app/**']`) + a
sibling package whose unique `paths`-resolved in-root file is NOT globbed by primary. Assert:
pruning stays OFF (primary source files ⊉ surface) AND the sibling-only symbol is still found.
Without this test, the naive glob-subset bug passes every loose-root and every references fixture
silently. Plus a loose-root fixture asserting all siblings pruned + byte-identical output.

## Backstop — genuinely-heavy find_usages / scope:all (depends t-000052)
Not Fix A's scope. process-mode isolation + per-engine `--max-old-space-size` + child-exit →
`ToolFailure{oom}`. The specced cross-workspace LRU governor does NOT catch one engine blowing
budget mid-op → scope as per-engine cap + child-death→ToolFailure. Longer-term for proper
monorepos: project-reference REDIRECTS (§9) to share lib/@types across programs (24 members
≈ 3.6 GB even without the flat root). Territory overlaps t-000052 → GO after it.
